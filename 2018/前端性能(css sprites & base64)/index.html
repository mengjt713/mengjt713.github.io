前端性能(css sprites&base64)

前情概要

近期打算好好写一些工作中遇到的一些东西，又想着高质量的文章，一直不知道如何开头，这个头要开的好，绞尽脑汁，前几天才遇到一个前端浏览器的性能问题，当断不断必自乱，就来搞下前端优化的一些东西好了，首先是关于图片的的处理，css 雪碧图和base64格式图片。都是通过减少http请求从而达到性能优化的目的，请求变少了，那么加载起来就变快了，自然看起来就很清爽了。

Css Sprites

CSSSprites在国内很多人叫css精灵，是一种网页图片应用处理方式。它允许你将一个页面涉及到的所有零星图片都包含到一张大图中去，这样一来，当访问该页面时，载入的图片就不会像以前那样一幅一幅地慢慢显示出来了。对于当前网络流行的速度而言，不高于200KB的单张图片的所需载入时间基本是差不多的，所以无需顾忌这个问题。 —— 百度百科

基本原理

将许多的小图整合到一张大图片中，利用css中的background-image属性，background-position属性定位某个图片位置，从而达到在大图片中引用某个部位的小图片的效果。

好处

  • 多图变一张图，减少了请求次数
  • 多图合并为一张图，总体大小<=多图大小之和
  • 命名规范化。一张图只需要命名一次，不用头疼起名称

缺点

-前期需要处理图片将小图合并，多了些工作量（还好有在线工具帮忙处理）
-对于需要经常改变的图片维护起来麻烦。

base64

图片生成base64可以用一些工具，如在线工具，但在项目中这样一个图片这样生成是挺繁琐。
特别说下，webpack中的url-loader可以完成这个工作，可以对限制大小的图片进行base64的转换，非常方便。

好处

  • base64的图片会随着html或者css一起下载到浏览器,减少了请求.
  • 可避免跨域问题

缺点

  • 低版本的IE浏览器不兼容。
  • 体积会比原来的图片大一点。
  • css中过多使用base64图片会使得css过大，不利于css的加载。

最后的最后

上面说的两个处理图片共同点是都是应用于小图片，都能减少请求数，但并不是所有的图片都适用，尺寸大的图片就不应使用。两种方式都有利有弊，应该适应场景使用，权衡利弊，方可将这两种方法能力发挥好。
